# Agent 工厂模式

先判断任务类型，再决定任务图和审批链的最小要求。

## 标准构建

适用场景：

- 新功能
- 新系统
- 多模块改动
- 多 agent 协作任务
- 需要明确任务图和并行策略的目标

执行顺序：

1. 任务建模
2. 能力边界
3. 任务图生成
4. Agent 分工
5. 执行
6. 评测审查
7. 风险审批
8. 发布与审计

规则：

- 先产出`任务规格`。
- 先定义能力边界，再生成`任务图`。
- 先完成`Agent 分工单`，再进入大规模执行。
- 涉及对外发布、不可逆变更、破坏性操作或高成本调用时，必须走`风险审批`。

## 修复模式

适用场景：

- 缺陷修复
- 小范围行为回归
- 已有清晰复现路径
- 不需要大规模重新拆图

执行顺序：

1. 任务建模
2. 能力边界
3. 任务图生成
4. 执行
5. 评测审查
6. 发布与审计

规则：

- 仍然先输出最小`任务规格`，不要直接开改。
- 可以把`任务图`压缩为最少必要节点。
- 可以省略完整`Agent 分工单`，但必须说明当前由哪个角色主控。
- 仍然必须输出`评测单`和`审计记录`。
- 如果修复过程中发现影响范围扩大，立即切回`标准构建`。

## 热修模式

适用场景：

- 生产阻塞
- 服务降级
- 数据错误扩散
- 必须先恢复系统可用性

执行顺序：

1. 任务建模
2. 执行
3. 评测审查
4. 风险审批
5. 发布与审计

规则：

- 先用 3 到 6 行写清故障现象、影响范围、临时目标和回滚抓手。
- 允许跳过完整任务图，但必须给出最小执行路径。
- 发布前必须有`风险闸门单`和`审计记录`。
- 热修完成后如果需要长期修复，回到`标准构建`补全任务图和分工。

## 模式切换

出现以下情况时中断当前模式并切换：

- 修复任务发现涉及多模块重构、数据迁移或多个角色联动：切换到`标准构建`。
- 标准构建进入生产窗口且出现阻塞事故：切换到`热修模式`。
- 热修后需要补长期治理或重构：回到`标准构建`。
