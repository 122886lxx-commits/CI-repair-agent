# Agent 系统评审与风险分级

在`评测审查`、`风险审批`和`发布与审计`阶段使用这份规则。

## 评审重点

优先检查这些问题：

- 任务分配错误导致上下文丢失或责任悬空
- Agent 权限、工具边界或审批边界不清
- 节点完成定义无法验证
- 不可恢复步骤缺少回滚设计
- 缺失评测或评测结果不足以支撑发布
- 有生产变更但无法形成追责链

## 风险分级

### P0

定义：

- 会导致生产不可用、数据破坏、安全越界、错误自动扩散或完全失控执行

处理：

- 立即阻塞执行或发布
- 优先给出止损、回滚和人工接管建议

### P1

定义：

- 关键任务节点没有完成定义
- 高风险动作未经过审批
- 主要执行路径存在明显失控风险

处理：

- 默认阻塞后续阶段
- 只有用户明确接受风险并记录审批，才允许继续

### P2

定义：

- 次要节点交接不清
- 局部评测不足
- 审计记录不完整但短期可补

处理：

- 不一定阻塞执行
- 必须记录补救动作和补单要求

### P3

定义：

- 文档表达不清
- 角色命名或状态描述可优化
- 低风险流程优化建议

处理：

- 不阻塞执行
- 放入后续改进

## 评测审查输出格式

发现问题时，按严重级别排序并使用下面格式：

```md
# 评测审查

## Findings
- [P1] 问题标题：影响、触发条件、为什么会破坏 agent 执行或发布
- [P2] 问题标题：影响、触发条件、为什么需要补强

## Blocking Conditions
- 

## Summary
- 
```

没有阻塞问题时，使用下面格式：

```md
# 评测审查

## Findings
- 未发现阻塞级问题。

## Residual Risks
- 

## Summary
- 
```

## 发布前检查

进入`发布与审计`前，至少确认这些点：

- `任务规格`和最终执行结果一致
- `任务图`或最小执行路径足以解释执行过程
- 关键评测已执行，未执行项已说明原因
- 需要审批的动作已经过`风险闸门单`
- `审计记录`包含发布摘要、回滚方式和关键决策
