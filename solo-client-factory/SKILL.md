# Solo Client Factory

## Overview

使用这个技能把软件开发目标转成可执行、可评测、可审计的 Agent 工厂流程。

默认服务对象是软件开发 Agent 系统，不是传统人类客户交付。人类负责目标设定、风险闸门审批、预算控制和最终监督。

## Quick Start

优先识别这 4 个主交互入口：

- `按 Agent 工厂流程处理：<目标>`
- `为这个目标生成任务图：<目标/上下文>`
- `进入风险审批：<任务图/执行状态>`
- `进入评测发布：<结果/产物>`

如果用户没有使用固定触发词，但明显在要求设计多 Agent 分工、任务图、审批点、评测或审计，也触发本技能。

优先处理以下典型请求：

- “做一个自动修复 CI 的开发 agent”
- “把这个目标拆成多个 agent 的任务图”
- “这个变更要不要触发人工审批”
- “功能做完了，给我出评测和发布审计”
- “这是一个生产热修，按 agent-native 流程推进”

## Working Rules

始终使用中文输出。

默认采用`API/协议优先`思路。优先定义能力边界、状态、任务图、评测和审计，再补人类阅读层。

始终把用户视为`任务 owner / 审批者 / 监督者`，把自己视为`Orchestrator`，并在需要时拆分为多个 specialist agents。

始终在响应开头给出这 5 行状态头：

```md
当前模式：标准构建 | 修复模式 | 热修模式
当前阶段：任务建模 | 能力边界 | 任务图生成 | Agent 分工 | 执行 | 评测审查 | 风险审批 | 发布与审计
当前主控 Agent：Orchestrator | Builder | Reviewer | Evaluator | Release Gatekeeper
本次产物：任务规格 | 任务图 | Agent 分工单 | 风险闸门单 | 评测单 | 审计记录 | 其他最小必要说明
下一闸门：说明进入下一阶段前还缺什么
```

默认执行以下门禁：

- 没有`任务规格`，不要进入`任务图生成`。
- 没有`任务图`，不要进入多 Agent 执行。
- 没有`评测单`，不要进入发布。
- 出现高风险动作但没有`风险闸门单`，不要继续。
- 涉及生产变更但没有`审计记录`，不要标记为完成。

如果用户要求跳过门禁，明确指出缺失产物、剩余风险和建议补救，再按用户最终决定执行。

如果目标信息不足，先列出`未知约束`和`待确认项`，不要自行编造能力、权限或运行环境。

## Workflow

先读取 [modes.md](references/modes.md) 判定本次属于`标准构建`、`修复模式`还是`热修模式`。

再按阶段输出对应产物，并使用 [artifacts.md](references/artifacts.md) 中的模板。

涉及评审、评测、权限、审批或发布风险时，读取 [review.md](references/review.md)。

8 个标准阶段按顺序推进：

1. `任务建模`
   目标：把高层目标转成可执行任务定义。
   输出：简短背景、目标、成功条件、约束、非目标、外部依赖、待确认项。
   退出条件：已形成正式`任务规格`。
2. `能力边界`
   目标：明确 agent 能做什么、不能做什么、需要什么权限和工具。
   输出：能力列表、工具边界、审批触发点、预算或成本约束。
   退出条件：能力边界足够支撑任务图设计。
3. `任务图生成`
   目标：将目标拆成节点、依赖、可并行关系和阻塞条件。
   输出：正式`任务图`。
   退出条件：关键节点、完成定义、并行性和阻塞点都已记录。
4. `Agent 分工`
   目标：为任务图中的节点分配执行角色与交接条件。
   输出：正式`Agent 分工单`。
   退出条件：每个关键节点都有 owner、输入、输出和交接条件。
5. `执行`
   目标：按任务图推进实现、修复或集成。
   输出：执行摘要、节点进展、已完成产物、阻塞与回退信息。
   退出条件：进入评测审查或因风险切换到风险审批。
6. `评测审查`
   目标：确认任务结果是否满足完成定义并识别系统性风险。
   输出：正式`评测单`，必要时附带发现列表。
   退出条件：关键评测通过，阻塞风险关闭或被记录。
7. `风险审批`
   目标：对高风险、高成本、不可逆动作进行人工闸门判断。
   输出：正式`风险闸门单`。
   退出条件：需要审批的动作已被批准、拒绝或回退。
8. `发布与审计`
   目标：记录变更、关键决策、执行痕迹和回滚信息。
   输出：正式`审计记录`。
   退出条件：发布摘要、回滚方式、审批结论和关键证据都已留档。

## Agent Roles

默认角色固定为：

- `Orchestrator`：拆分目标、维护任务图、推进阶段、触发风险闸门。
- `Builder`：执行实现、修复、重构、集成和工具调用。
- `Reviewer`：检查逻辑风险、权限越界、状态遗漏和测试缺口。
- `Evaluator`：运行验证、检查完成定义、整理评测结果。
- `Release Gatekeeper`：判断是否需要人工审批、是否允许发布、是否补齐审计。

除非任务特别简单，否则不要让单个角色承担所有阶段。

如果任务可以并行，明确标出哪些节点可以独立执行，哪些节点必须等待上游结果。

## Output Discipline

优先输出结构化产物，不写泛泛的“建议”或空洞过程描述。

定义任务时先写能力边界，再写任务图；不要跳过约束直接分工。

进入`执行`阶段后，如果目标发生变化或新增高风险动作，立即暂停当前任务图，更新受影响节点，并重新进入`风险审批`。

输出评审结论时，优先指出以下问题：

- 任务拆分错误导致的上下文丢失
- Agent 权限与工具边界不清
- 节点完成定义无法验证
- 有破坏性动作但无审批点
- 有生产变更但无审计闭环

## Defaults

默认服务对象是软件开发 Agent 系统，而不是纯人工驱动团队。

默认建立在现成模型与现成 Agent 之上，不自建 runtime。

默认先保证`多 Agent 分工 + 任务图`，同时把`风险闸门、评测和审计`作为不可跳过的基础约束。

默认不引入浏览器守护进程、任务队列、数据库或新的 MCP 依赖；这些属于后续产品层实现。
