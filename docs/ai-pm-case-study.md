# AI 产品经理 Case Study

## 项目名称

CI Repair Agent: 面向开发团队与开发 Agent 的 CI 故障修复系统

## 一句话定义

这是一个面向 `GitHub + GitHub Actions` 的 agent-native 工具：当 CI 失败时，系统自动读取失败上下文、生成低风险修复方案、在隔离环境中验证、按风险决定是自动创建 PR 还是进入人工审批，并保留完整审计记录。

它不是一个通用代码助手，而是一个面向工程系统的 `自动执行 + 风险治理` 产品。

---

## 1. 背景与问题

### 我观察到的问题

在团队开发中，CI 失败是非常高频但非常碎片化的一类问题：

- 开发者需要手动查看 workflow 日志、定位失败原因、尝试修复、重新跑验证
- 很多失败本质上是低风险问题，比如类型错误、测试断言、配置偏差，但仍然需要人工介入
- 现有 AI 编码工具能生成 patch，但缺少 `风险闸门、评测、审计`，很难直接进入团队生产流程

### 更本质的问题

AI 时代真正缺的不是“能写代码的模型”，而是：

`如何让 agent 在工程系统中安全、可控、可追责地执行任务`

所以我没有把这个产品定义成“自动修 Bug 工具”，而是把它定义成：

`一个针对 CI 修复场景的 Agent 执行控制面`

---

## 2. 目标用户

### 直接用户

- 开发者
- Tech Lead / Reviewer
- 平台工程师
- 小团队创始人或一人公司技术负责人

### 实际执行主体

- Orchestrator Agent
- Builder Agent
- Reviewer Agent
- Evaluator Agent
- Release Gatekeeper Agent

### 为什么是这批用户

因为这个产品服务的不是普通终端用户，而是：

- 有 GitHub Actions
- 有 CI 故障处理成本
- 愿意让 Agent 参与修复
- 但又不能接受 Agent 无约束执行

换句话说，它优先服务的是：

`正在从“人写代码”过渡到“人+Agent 共建软件”的团队`

---

## 3. 核心场景

### 典型场景

一个 PR 合入后，CI 失败：

1. 系统接收到 workflow run 失败事件
2. 读取失败步骤、日志、关联 commit
3. 生成任务规格和任务图，而不是直接改代码
4. Builder 生成 patch
5. Evaluator 在隔离环境验证 patch
6. Reviewer 和规则系统判断风险
7. 如果是低风险改动，自动创建 draft PR
8. 如果涉及高风险动作，例如 workflow、迁移、权限文件，则进入人工审批
9. 全过程生成审计记录，支持 rerun 和回溯

### 为什么先从 CI 修复切入

我没有直接做“通用开发 Agent 平台”，因为那样范围太大，难以验证。

CI 修复是一个很好的切口，因为它同时具备：

- 高频问题
- 明确输入：workflow run、日志、commit
- 明确输出：修复 patch、评测结果、PR
- 可量化结果：是否修复、是否通过验证、是否需要审批
- 高治理价值：非常适合放入风险闸门和审计机制

---

## 4. 产品定义

### 产品目标

把 CI 故障修复从“人工排查流程”变成“可自动执行、可审批、可审计的系统流程”。

### 我定义的产品边界

#### v1 做什么

- 只支持 `GitHub + GitHub Actions`
- 只支持单租户内部使用
- 只支持独立分支 + draft PR
- 支持风险分级、人工审批、评测和审计

#### v1 不做什么

- 不做自动合并
- 不直接 push 主分支
- 不做多租户
- 不做通用 CI 平台抽象
- 不做自建 agent runtime

### 这个边界为什么重要

因为 AI 产品最容易犯的错误是：

`把“能力可能性”误当成“产品边界”`

我刻意把能力收窄到一个高价值、可验证、可治理的场景里，先验证：

- 用户是否真的愿意用
- 自动修复是否能减少人工排查
- 风险闸门是否能建立信任

---

## 5. 核心产品设计

### 5.1 Agent-native 工作流

我把流程拆成 8 个阶段：

1. 任务建模
2. 能力边界
3. 任务图生成
4. Agent 分工
5. 执行
6. 评测审查
7. 风险审批
8. 发布与审计

这和传统“提交工单 -> 修 bug”不同。  
我把它设计成一个 Agent-first 的执行系统，而不是人类操作界面优先的系统。

### 5.2 多 Agent 分工

我定义了 5 个固定角色：

- `Orchestrator`：负责拆任务、推进状态、汇总输出
- `Builder`：负责生成 patch
- `Reviewer`：负责识别越权改动、逻辑风险、范围扩张
- `Evaluator`：负责跑验证、检查完成定义
- `Release Gatekeeper`：负责判断是否允许发布或必须审批

这样做的原因是：

- 让系统从一开始就适配多 Agent 协作
- 让不同职责可以被审计
- 让“执行”和“放行”不是同一个角色

### 5.3 风险闸门

这是我认为这个项目最关键的产品设计。

高风险动作必须审批，例如：

- 修改 `.github/workflows/*`
- 基础设施配置
- 权限和认证相关文件
- 数据库迁移
- 删除操作
- 外部写调用
- 依赖大版本升级

低风险动作才允许自动执行：

- 创建分支
- 应用 patch
- 跑验证
- 创建 draft PR

这里我想解决的不是“能不能自动修”，而是：

`在什么条件下，用户敢让它自动修`

### 5.4 评测与审计

我把 `评测单` 和 `审计记录` 设计成强制产物，不是附加信息。

因为一旦进入真实工程流程，系统必须回答 3 个问题：

- 它为什么这么改
- 它怎么证明自己没改错
- 如果出了问题，能不能追责和回滚

所以我把以下对象做成系统核心数据结构：

- `TaskSpec`
- `TaskGraph`
- `AgentAssignment`
- `ApprovalGate`
- `EvalResult`
- `AuditEvent`

## 6. MVP 方案

### 产品形态

我把产品拆成 5 个子系统：

- `API / Control Plane`
- `Worker / Orchestrator`
- `Sandbox Runner`
- `GitHub Integration`
- `Web Console`

### 为什么这样拆

因为我希望它具备真实产品的最小闭环，而不是一个 prompt demo：

- API 负责接任务、查状态、审批
- Worker 负责推进阶段状态机
- Sandbox 负责隔离验证
- GitHub 集成负责 workflow 和 PR
- Console 负责给人类审阅和审批

### 控制台 MVP

控制台只做 4 个页面：

- 任务列表
- 任务详情
- Diff 与审批
- 审计时间线

我刻意没有先做复杂前端，因为核心价值不在“交互华丽”，而在：

- 风险是否识别正确
- 审批链是否清楚
- 结果是否可信

## 7. 我做过的关键取舍

### 取舍 1：先做窄场景，不做通用开发 Agent

原因：

- 通用开发 Agent 很难定义成功标准
- CI 修复有清晰输入、输出和验证机制
- 更适合做 PM 驱动的 MVP

### 取舍 2：先做 draft PR，不做自动 merge

原因：

- 自动 merge 会大幅提升信任门槛
- 草案 PR 已经能验证“修复建议是否有用”
- 这能保留人工监督，同时减少人工排查时间

### 取舍 3：规则系统和模型双保险

原因：

- 纯模型判断风险不够稳定
- 纯规则系统又没有修复能力
- 所以我把风险拦截交给规则兜底，把修复提议交给模型

### 取舍 4：先做单租户内部系统

原因：

- 多租户、RBAC、计费不是当前核心验证点
- 先证明“是否真的能降低 CI 故障处理成本”更重要

## 8. 我作为 PM 的角色

如果从 PM 角度讲，这个项目里我强调的不是“我写了多少代码”，而是：

### 8.1 问题定义

我明确把问题定义成：

`AI 在软件工程流程中的可控执行`

而不是简单的“自动修 bug”。

### 8.2 用户与边界

我清楚区分了：

- 谁是操作者
- 谁是执行者
- 谁是审批者
- 哪些动作能自动化
- 哪些动作必须保留人工

### 8.3 工作流设计

我没有把它做成单轮 prompt，而是设计成了：

- 有阶段
- 有状态机
- 有审批门槛
- 有恢复机制
- 有审计痕迹

### 8.4 风险治理

我优先设计了信任机制，而不是炫自动化能力。

这件事体现的是 PM 判断：

`用户不会因为“更自动”而买单，用户会因为“自动但可控”而愿意试用`

## 9. 如何讲这个项目

### 90 秒版本

我做了一个面向开发团队的 AI-native 工程产品，核心场景是自动修复 GitHub Actions 失败。

这个项目的重点不是生成 patch，而是让 Agent 修复流程可以进入生产：系统会先把一次 CI 故障建模成任务规格和任务图，再分配给不同 Agent 去生成修复、做评测、识别风险。如果是低风险改动，就自动创建 draft PR；如果涉及 workflow、权限、迁移等高风险内容，就必须进入人工审批。整个过程保留评测结果和审计记录。

我把这个项目当成一个 AI PM 案例来做，因为它本质上解决的是 Agent 产品里最核心的几个问题：任务拆解、执行控制、风险闸门、评测与审计。

### 3 分钟版本

我观察到一个很现实的问题：AI 虽然已经能写 patch，但在团队生产流程里，大家不敢直接放它执行。问题不在模型能力，而在缺少治理层。所以我没有做一个通用代码助手，而是选了一个非常具体、非常适合验证的场景：CI 失败修复。

这个场景输入非常清楚，有 workflow run、日志和 commit；输出也很清楚，是 patch、验证结果和 PR。于是我围绕这个场景定义了一个 Agent-native 工作流，把整个流程拆成任务建模、任务图、Agent 分工、执行、评测、风险审批和审计。核心设计是把高风险动作和低风险动作分开，高风险必须审批，低风险才自动创建 draft PR。

对我来说，这个项目最重要的不是工程实现本身，而是它体现了我对 AI 产品的理解：未来很多软件不是服务终端用户，而是服务 Agent 系统，而这类产品的核心不是 UI，而是流程控制、权限边界、评测和信任机制。

## 10. 面试官可能会追问的问题

### Q1：为什么这是 PM 项目，不是纯工程项目？

因为它的核心价值不是“我实现了 GitHub 接口”，而是我完成了：

- 用户定义
- 场景聚焦
- 边界控制
- 风险治理
- 状态机设计
- MVP 取舍

工程只是承载这些产品决策的方式。

### Q2：为什么先从 CI 修复做，而不是更大的 Agent 平台？

因为 CI 修复有更强的验证性和更清晰的 ROI。  
它能更快判断：

- 自动修复是否真的有用
- 用户是否愿意接受 Agent 输出
- 风险闸门是否能建立信任

### Q3：这套系统最大的 moat 是什么？

不是模型本身，而是：

- 风险规则
- 评测体系
- 审计与恢复机制
- 和真实工程流程的集成深度

### Q4：如果继续做下一版，会优先做什么？

我会优先做三件事：

1. 基于真实失败样本建立评测集
2. 补齐更细的风险规则和恢复策略
3. 验证从 draft PR 到团队采纳率的完整闭环

## 11. 简历写法

### 简历一句话版本

设计并实现一套面向 GitHub Actions 的 agent-native CI 修复系统，将故障修复流程产品化为 `任务建模-修复-评测-审批-审计` 闭环，重点解决 AI 在工程系统中的可控执行问题。

### 简历要点版本

- 定义 CI 故障修复场景下的 Agent-native 工作流，拆分 `Orchestrator / Builder / Reviewer / Evaluator / Gatekeeper` 多 Agent 协作模型
- 设计高风险动作审批机制，将 workflow、权限、迁移等改动纳入人工闸门，低风险修复自动创建 draft PR
- 设计 `TaskSpec / TaskGraph / EvalResult / AuditEvent` 等核心对象，建立评测、审计与 rerun 恢复能力
- 以单租户 MVP 验证 AI 工程产品在真实研发流程中的可控自动化价值

## 12. 我从这个项目得到的 PM 结论

这个项目让我更明确一件事：

AI 产品不应该只追求“让模型更能做事”，更重要的是：

`让模型在一个可信的系统里做事`

对未来很多 AI-native 产品来说，真正的产品壁垒不只是模型调用，而是：

- workflow 设计
- risk gating
- eval
- audit
- recovery

这也是我认为 AI PM 会越来越重要的地方。
